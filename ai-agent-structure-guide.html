📦 Pattern 2 — Monorepo with Hierarchical AGENTS.md
ℹ️

The OpenAI repo uses 88 AGENTS.md files across its monorepo. Each package overrides root defaults. The agent reads the nearest file up the tree — closest wins, outer provides defaults.

monorepo/ ├── AGENTS.md ← Root defaults: stack, pnpm, shared conventions ├── CLAUDE.md ← Points to ./AGENTS.md ├── README.md ├── packages/ │ ├── api/ │ │ ├── AGENTS.md ← API-specific: GraphQL, Prisma, REST conventions │ │ └── src/ │ ├── web/ │ │ ├── AGENTS.md ← Frontend: React, Tailwind, component patterns │ │ └── src/ │ ├── worker/ │ │ ├── AGENTS.md ← Worker-specific: queue, retry, idempotency rules │ │ └── src/ │ └── shared/ │ ├── AGENTS.md ← Shared lib conventions │ └── src/ └── docs/ ├── ARCHITECTURE.md └── patterns/ ← AI pattern library, referenced from AGENTS.md
⚠️

Don't overload any level. Root AGENTS.md = global defaults only. Package AGENTS.md = what's unique to that subproject. Agents see all merged files in context — keep each focused.

🤖 Pattern 3 — Advanced ACS (Agent Context System) Layout

For larger projects with multiple specialized subagents, a dedicated .agents/ folder uses 3-tier progressive disclosure to keep context windows lean.

project/ ├── AGENTS.md ← Entry point, references .agents/ for depth ├── CLAUDE.md ├── .agents/ ← ACS root │ ├── acs.yaml ← Manifest: what skills/agents are available │ ├── context/ ← What agents need to KNOW (loaded on demand) │ │ ├── stack.md │ │ ├── architecture.md │ │ └── data-model.md │ ├── skills/ ← What agents can DO (activated by task match) │ │ ├── deploy.md │ │ ├── migrate-db.md │ │ └── review-pr.md │ ├── agents/ ← Named subagents with their own system prompts │ │ ├── backend-agent.md │ │ ├── frontend-agent.md │ │ └── test-agent.md │ └── permissions/ ← What agents can/cannot touch │ └── boundaries.md └── src/

Rule File Deep Dive
Each AI tool has its own format, hierarchy, and loading behavior. Here's everything you need to know about each one — and how they relate.
🟢 AGENTS.md — The Open Standard

What it is

Created by OpenAI in August 2025. Donated to Linux Foundation AAIF in December 2025. Simple Markdown, no special schema, no YAML frontmatter required. Supported by 20+ tools as of March 2026.

Hierarchy rules

  • 📍
    Place one at repo root for global defaults
  • 📦
    Place one inside each subpackage to override
  • 🔼
    Nearest file up the tree wins — closest to the file being edited takes precedence
  • ✍️
    User prompt always overrides file instructions

Sections to include

  • 📋
    Project overview — stack, language, key frameworks
  • ⚙️
    Commands — exact shell commands (build, test, lint, migrate)
  • 🎨
    Code style — only non-inferable conventions
  • 🧪
    Testing requirements — what to run, how to run it
  • 🚫
    Boundaries — never do X, always ask before Y
  • 📎
    Pointers — link to extended docs in docs/
🔵 CLAUDE.md — Claude Code Specific

Hierarchy (bottom-up)

  • 🏢
    Enterprise policy — loads first (broadest)
  • 👤
    ~/.claude/CLAUDE.md — your personal preferences
  • 📁
    Project root CLAUDE.md — project-level
  • 📂
    Subdirectory CLAUDE.md — overrides for that folder

More specific instructions override broader ones. Each level adds, not replaces.

Claude Code extras

  • 📥
    @imports — compose instructions from multiple files: @docs/patterns.md
  • 🔪
    Slash commands — stored in .claude/commands/
  • 🧠
    Skills — stored in .claude/skills/ (SKILL.md files)
  • 🤖
    Subagents — defined with own system prompts and tool access
⚠️

Claude Code injects CLAUDE.md with a system reminder: "this context may or may not be relevant to your tasks." This means Claude will ignore instructions it deems irrelevant. Implication: keep the file universally applicable and under 300 lines.

🟣 GEMINI.md — Google Gemini CLI

Google's Gemini CLI reads GEMINI.md first, falls back to AGENTS.md. Same hierarchical nearest-file logic. Use GEMINI.md for Gemini-specific settings (model selection, grounding options, tool permissions). Point the bulk of your rules to AGENTS.md.

GEMINI.md
# GEMINI.md # This file is Gemini CLI-specific. # All project conventions are in AGENTS.md. ## Gemini-Specific Settings - Model: gemini-2.5-pro - Ground responses using Google Search for API docs - Do NOT use code execution tool for DB migrations ## Shared Rules See ./AGENTS.md for all project-level conventions.
🟠 .cursorrules & Cursor Instructions

Cursor evolved from a single .cursorrules to scoped .cursor/rules/*.mdc files. Each rule can be Always On, Manual (via @mention), or Auto (model decides). Limits: 6,000 chars per rule, 12,000 chars total.

Scoped rules (glob frontmatter)

--- description: TypeScript React rules globs: ["**/*.tsx", "**/*.ts"] alwaysApply: false --- Use functional components with TypeScript interfaces. Always use server components unless client interactivity is required.
💡

Use Cursor-scoped rules for file-type specific instructions. Put all universal project rules in AGENTS.md so they're available to every tool.

🟡 GitHub Copilot Instructions

Copilot reads .github/copilot-instructions.md. Since July 2025 it also supports .github/instructions/*.instructions.md with glob-pattern frontmatter — same scoped approach as Cursor. Also natively reads AGENTS.md.

--- applyTo: "**/*.tsx" --- Use functional components with TypeScript interfaces. Always use server components unless client interactivity is required.

AGENTS.md Template Library
Production-ready templates for different project types. Copy, adapt, and keep under 150–300 lines.
⚠️

Instruction Budget: Frontier thinking LLMs follow ~150–200 instructions reliably. Claude Code's system prompt already uses ~50. Your AGENTS.md should use as few as needed — every line competes for attention with actual work.

🤝 Subagent Orchestration Pattern

For large projects, route tasks to specialized agents — each gets a smaller, focused context. Claude Code supports defining subagents with their own system prompts and tool access.

.agents/agents/ ├── orchestrator.md ← Routes tasks to subagents ├── backend-agent.md ← DB, API, auth logic ├── frontend-agent.md ← React, UI, accessibility ├── test-agent.md ← Write & run tests only └── review-agent.md ← Code review & security
⚠️

Start with 2–3 parallel agents max. Ensure tasks are truly independent — never two agents writing to the same file at once.

AI-Readable Documentation System
AGENTS.md is just the entry point. A well-designed docs/ system gives agents the deep context they need without clogging the rule file. Reference from AGENTS.md; load on demand.
ℹ️

LLMs are in-context learners. If your code follows a consistent pattern, a few examples (or a well-written pattern doc) teach the agent faster than listing rules. Reference real files in the repo as concrete examples — agents can read them.

📚 Recommended docs/ Structure
docs/ ├── ARCHITECTURE.md ← System design, data flow, service boundaries ├── API_CONVENTIONS.md ← REST/GraphQL patterns, error formats, versioning ├── LLM_CONVENTIONS.md ← Prompt patterns, caching rules, token budgets ├── SECURITY.md ← Auth flows, data handling, what agents must NOT do ├── TESTING.md ← Test strategy, mocking approach, fixture usage └── patterns/ ← Code pattern library (anti-patterns + correct examples) ├── components-button.md ├── components-form.md ├── server-functions.md ├── error-handling.md ├── database-queries.md └── CONTRIBUTE.md ← How to add patterns
📋 Pattern File Structure (Each file in docs/patterns/)
❌ Anti-pattern
## Button Anti-patterns // Wrong: class component class Button extends React.Component { render() { return <button>Click</button> } } // Wrong: default export export default Button // Wrong: inline styles <button style={{color: 'red'}}>
✅ Correct pattern
## Button Patterns // Correct: functional + named export interface ButtonProps { label: string onClick: () => void variant?: 'primary' | 'ghost' } export const Button = ({ label, onClick, variant = 'primary' }: ButtonProps) => ( <button className={cn(styles[variant])}> {label} </button> ) // See: src/components/ui/Button.tsx

Key pattern file elements: structural trees showing component composition, Wrong vs Correct comparisons, references to real files in the repo as concrete examples.

What NOT To Do
The most common mistakes that hurt agent performance — backed by the ETH Zurich AGENTbench research and practitioner experience.
🚫

Never auto-generate AGENTS.md. LLM-generated files are comprehensive but not focused. The ETH Zurich study found they can hurt performance vs no file at all. Write yours by hand — every line must earn its place.

🚫

Never duplicate across tool files. Don't copy-paste the same rules into CLAUDE.md, .cursorrules, and copilot-instructions.md. Maintain AGENTS.md as the single source of truth; have tool-specific files reference it.

🚫

Never put things a linter handles. "Use 2-space indentation" or "always add trailing commas" are Prettier's job, not your instruction budget's. Same for TypeScript — if the type system enforces it, don't repeat it in AGENTS.md.

🚫

Never include task-specific instructions in AGENTS.md. "When building the payment module, make sure to..." belongs in your prompt or a SPEC file, not a persistent config that loads in every session.

🚫

Never let the file go stale. A wrong instruction is worse than no instruction. When your stack changes, update AGENTS.md in the same PR. Add AGENTS.md review to your PR checklist.

🚫

Never include obvious things. "This project uses JavaScript", "Write clean code", "Follow best practices" — agents can infer these from package.json and the codebase. Every obvious line crowds out actually useful instructions.

🚫

Never paste full API docs. Link to them instead. Use a docs/ directory for supplementary context that agents can pull in when needed, rather than embedding it all in AGENTS.md.

🚫

Never run multiple agents on the same file. Last-write-wins conflicts are painful. Ensure parallel agent workstreams are truly independent — separate branches, separate file domains.

✅ Quick Self-Check Before Publishing Your AGENTS.md
  • ✓
    Is every instruction universally applicable to all tasks in this project? (If not, move to a SPEC or prompt)
  • ✓
    Are all commands wrapped in backticks and copy-paste runnable? (Exact commands, not descriptions)
  • ✓
    Is the file under 300 lines? Under 150 is better.
  • ✓
    Have you removed anything that a linter, formatter, or type system already enforces?
  • ✓
    Have you removed anything inferable from the codebase itself?
  • ✓

  • ✓

  • ✓

  • ✓

  • ✓
